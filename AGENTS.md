# Codex 작업 규칙

## 기본 원칙

이 프로젝트는 학습과 면접 포트폴리오 목적의 React + TypeScript + Node.js 기반 웹 서비스입니다.

프로젝트 진행 폴더는 다음 폴더 아래와 같습니다 Feedback-2-Backlog 해당 폴더 내부에서만 수정사항을 변경하십시오

코드를 작성할 때는 단순히 동작하는 코드만 만들지 말고, 사용자가 코드 흐름을 이해할 수 있도록 명확한 구조와 읽기 쉬운 이름을 우선합니다.

## 작업 전 확인 규칙

다음 상황에서는 바로 수정하지 말고 반드시 먼저 사용자에게 확인합니다.

1. 여러 파일을 동시에 크게 수정해야 하는 경우
2. 폴더 구조를 바꾸거나 파일을 대량 생성/삭제해야 하는 경우
3. 기존 기능의 동작 방식이 바뀌는 경우
4. 사용자의 요구사항이 애매하거나 여러 방식으로 해석될 수 있는 경우
5. 기술 선택지가 여러 개 있는 경우
6. 현재 코드 의도를 정확히 이해하지 못한 경우

특히 사용자의 의도를 추론해서 임의로 구현하지 않습니다.
이해가 부족하면 “제가 이해한 내용은 A인데, B로 진행해도 될까요?”처럼 먼저 확인합니다.

## 수정 방식

작업은 가능한 한 작은 단위로 나누어 진행합니다.

한 번에 전체를 완성하려 하지 말고 다음 순서를 따릅니다.

1. 현재 코드 구조 파악
2. 필요한 변경 사항 요약
3. 수정 대상 파일 목록 제시
4. 사용자 확인이 필요한 경우 확인 요청
5. 코드 수정
6. 수정 후 실행 방법과 확인 포인트 설명
7. 작업이 끝나면 주석을 추가한다.

## 코드 작성 규칙

* React 컴포넌트는 역할이 명확하게 드러나는 이름을 사용합니다.
* TypeScript 타입을 가능하면 명시합니다.
* API 요청, 데이터 타입, 페이지 컴포넌트는 분리 가능한 구조로 작성합니다.
* 유지보수가 편하게 파일들을 분리하여 작성합니다.
* 처음부터 과한 추상화는 하지 않습니다.
* MVP 구현을 우선하고, 로그인/권한/배포/복잡한 상태관리 등은 사용자가 요청하기 전까지 추가하지 않습니다.
* 불필요한 라이브러리를 임의로 추가하지 않습니다.
* 라이브러리 추가가 필요하면 먼저 이유를 설명하고 확인받습니다.

## 프로젝트 목표

이 프로젝트는 Feedback2Backlog입니다.

서비스 흐름은 다음과 같습니다.

소비자:

* 피드백 작성
* 내 피드백 처리 상태 확인

개발자/운영자:

* 전체 피드백 확인
* 미처리 피드백을 하루 단위로 백로그 자동 생성
* 생성된 백로그 확인
* 백로그 해결 완료 처리
* 연결된 피드백 상태를 자동으로 처리 완료로 변경

## 응답 방식

코드를 수정하기 전에는 다음 형식으로 답합니다.

* 이해한 요구사항
* 수정할 파일
* 구현 방식
* 확인이 필요한 부분

코드를 수정한 후에는 다음 형식으로 답합니다.

* 수정한 파일
* 구현된 기능
* 실행 방법
* 확인해야 할 화면 또는 버튼
* 다음 단계 제안

## 금지 사항

* 사용자의 의도를 임의로 추론해서 큰 구조를 바꾸지 않습니다.
* 한 번에 너무 많은 파일을 수정하지 않습니다.
* 사용자가 요청하지 않은 사항들을 추가하지 않습니다.
* 기존 코드 스타일을 불필요하게 전부 갈아엎지 않습니다.
* 에러가 발생했는데 원인을 모르는 상태로 임의 수정하지 않습니다.
* “아마 이럴 것이다”라고 판단해서 진행하지 않습니다.
