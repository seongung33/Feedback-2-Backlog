# Feedback2Backlog 프로젝트 명세서

## 1. 프로젝트 개요

**Feedback2Backlog**는 사용자가 작성한 서비스 피드백을 개발자/운영자가 하루 단위로 모아 확인하고, 미처리 피드백을 RAG 기반으로 분석하여 제품 백로그로 자동 전환하는 웹 서비스이다.

소비자는 단순히 피드백을 작성하고 자신의 피드백 처리 상태를 확인한다. 개발자/운영자는 전체 피드백을 확인하고, 아직 백로그로 전환되지 않은 피드백들을 묶어 백로그로 생성한다. 이후 생성된 백로그를 해결 완료 처리하면, 해당 백로그와 연결된 피드백들의 상태가 자동으로 처리 완료로 변경된다.

이 프로젝트는 단순 CRUD 서비스가 아니라, 고객 피드백을 제품 개선 작업으로 연결하는 흐름을 구현하는 것을 목표로 한다.

---

## 2. 프로젝트 목적

이 프로젝트의 목적은 다음과 같다.

1. 고객 피드백을 단순히 수집하는 것에서 끝내지 않고, 제품 개선 작업으로 전환하는 흐름을 구현한다.
2. 소비자 화면과 개발자/운영자 화면을 분리하여 실제 서비스 관점의 사용자 흐름을 설계한다.
3. 미처리 피드백을 하루 단위로 모아 백로그로 자동 생성하는 기능을 구현한다.
4. RAG 구조를 활용하여 기존 피드백, 기존 백로그, 제품 맥락을 참고한 백로그 생성 흐름을 설계한다.
5. React, TypeScript, Node.js 기반의 풀스택 웹 서비스 구조를 학습하고 포트폴리오로 정리한다.

---

## 3. 문제 정의

서비스 운영 과정에서 고객 피드백은 여러 경로를 통해 지속적으로 들어온다. 그러나 피드백이 단순 문의나 불만 사항으로만 남고, 실제 제품 백로그로 정리되지 않으면 제품 개선으로 이어지기 어렵다.

특히 초기 스타트업이나 작은 개발 조직에서는 다음과 같은 문제가 발생할 수 있다.

* 고객 피드백이 여러 곳에 흩어져 관리된다.
* 반복적으로 등장하는 고객 문제를 빠르게 파악하기 어렵다.
* 피드백을 개발 백로그로 변환하는 과정이 수작업에 의존한다.
* 어떤 피드백이 실제로 처리되었는지 고객에게 명확히 전달하기 어렵다.
* 개발자는 고객 문제보다 기능 단위 작업만 보게 되어 제품 관점의 우선순위 판단이 어려워질 수 있다.

Feedback2Backlog는 이러한 문제를 해결하기 위해 고객 피드백을 자동으로 분석하고, 유사한 피드백을 묶어 제품 백로그로 전환하는 기능을 제공한다.

## 4. 핵심 사용자

### 4.1 소비자

소비자는 서비스를 사용하는 일반 사용자이다.

소비자는 다음 기능만 사용한다.

* 피드백 작성
* 내가 작성한 피드백 목록 확인
* 피드백 처리 상태 확인

소비자는 백로그, 우선순위, 담당 영역, RAG 분석 결과와 같은 내부 개발 정보를 직접 보지 않는다.

### 4.2 개발자 / 운영자

개발자 또는 운영자는 고객 피드백을 제품 개선 작업으로 전환하는 사용자이다.

개발자/운영자는 다음 기능을 사용한다.

* 전체 피드백 목록 확인
* 미처리 피드백 확인
* 미처리 피드백 기반 백로그 자동 생성
* 생성된 백로그 목록 확인
* 백로그 상태 변경
* 백로그 해결 완료 처리
* 백로그와 연결된 피드백 상태 자동 변경 확인

## 5. 핵심 서비스 흐름

### 5.1 소비자 흐름

1. 소비자가 피드백 작성 화면에 접속한다.
2. 서비스 사용 중 불편했던 점과 개선되었으면 하는 점을 작성한다.
3. 피드백을 제출한다.
4. 제출된 피드백은 `pending` 상태로 저장된다.
5. 소비자는 내 피드백 목록에서 피드백 처리 상태를 확인한다.

### 5.2 개발자/운영자 흐름

1. 개발자/운영자가 관리자 화면에 접속한다.
2. 전체 피드백 목록에서 아직 처리되지 않은 피드백을 확인한다.
3. 하루 단위 또는 필요 시점에 `[미처리 피드백으로 백로그 생성]` 버튼을 클릭한다.
4. 시스템은 `pending` 상태의 피드백들을 가져온다.
5. 유사한 피드백끼리 묶는다.
6. 기존 피드백, 기존 백로그, 제품 맥락을 참고하여 백로그를 생성한다.
7. 생성된 백로그와 관련 피드백들이 연결된다.
8. 관련 피드백의 상태는 `converted`로 변경된다.
9. 개발자가 백로그를 해결하면 `[해결 완료]` 버튼을 클릭한다.
10. 백로그 상태는 `done`으로 변경된다.
11. 해당 백로그와 연결된 피드백들의 상태는 `resolved`로 변경된다.
12. 소비자는 내 피드백 목록에서 처리 완료 상태를 확인할 수 있다.

## 6. 화면 구성

## 6.1 소비자 화면

### `/feedback` - 피드백 작성 화면

사용자가 서비스에 대한 피드백을 작성하는 화면이다.

주요 요소:

* 서비스/기능 입력
* 불편했던 점 입력
* 개선되었으면 하는 점 입력
* 피드백 제출 버튼

주요 동작:

* 사용자가 피드백을 작성하고 제출한다.
* 제출된 피드백은 `pending` 상태로 저장된다.

---

### `/my-feedbacks` - 내 피드백 목록 화면

사용자가 자신이 작성한 피드백의 처리 상태를 확인하는 화면이다.

주요 요소:

* 피드백 내용
* 제출일
* 처리 상태

피드백 상태:

* `pending`: 접수됨
* `converted`: 백로그 반영됨
* `resolved`: 처리 완료

---

## 6.2 개발자/운영자 화면

### `/admin` - 메인 대시보드

개발자/운영자가 전체 현황을 요약해서 확인하는 화면이다.

주요 요소:

* 전체 피드백 수
* 미처리 피드백 수
* 생성된 백로그 수
* 완료된 백로그 수
* 최근 들어온 피드백
* 최근 생성된 백로그

---

### `/admin/feedbacks` - 전체 피드백 목록 화면

개발자/운영자가 모든 사용자 피드백을 확인하는 화면이다.

주요 요소:

* 피드백 내용
* 관련 기능
* 제출일
* 처리 상태
* 연결된 백로그 ID
* 미처리 피드백 기반 백로그 생성 버튼

주요 동작:

* `[미처리 피드백으로 백로그 생성]` 버튼을 클릭하면 `pending` 상태의 피드백들이 백로그로 전환된다.

---

### `/admin/backlogs` - 백로그 목록 화면

RAG 기반으로 생성된 백로그를 확인하고 상태를 변경하는 화면이다.

주요 요소:

* 백로그 제목
* 문제 요약
* 요구사항
* 우선순위
* 담당 영역
* 연결된 피드백 개수
* 백로그 상태
* 해결 완료 버튼

주요 동작:

* 개발자가 `[해결 완료]` 버튼을 클릭하면 해당 백로그 상태가 `done`으로 변경된다.
* 연결된 피드백들의 상태가 `resolved`로 자동 변경된다.

## 9. 데이터 모델

## 9.1 Feedback

사용자 피드백 데이터이다.

```ts
type FeedbackStatus = "pending" | "converted" | "resolved";

type Feedback = {
  id: number;
  userId: string;
  feature: string;
  painPoint: string;
  suggestion: string;
  content: string;
  status: FeedbackStatus;
  backlogId: number | null;
  createdAt: string;
};
```

필드 설명:

* `id`: 피드백 고유 ID
* `userId`: 작성자 ID
* `feature`: 관련 서비스 또는 기능
* `painPoint`: 사용자가 불편했던 점
* `suggestion`: 개선되었으면 하는 점
* `content`: 피드백 전체 내용
* `status`: 피드백 처리 상태
* `backlogId`: 연결된 백로그 ID
* `createdAt`: 작성일

### Feedback 상태값

```txt
pending   : 접수됨 / 아직 백로그로 전환되지 않음
converted : 백로그에 반영됨
resolved  : 연결된 백로그가 해결 완료됨
```

---

## 9.2 Backlog

개발자/운영자용 백로그 데이터이다.

```ts
type BacklogStatus = "todo" | "in_progress" | "done";
type BacklogPriority = "High" | "Medium" | "Low";

type Backlog = {
  id: number;
  title: string;
  problem: string;
  requirement: string;
  priority: BacklogPriority;
  area: string;
  status: BacklogStatus;
  relatedFeedbackIds: number[];
  createdAt: string;
  resolvedAt: string | null;
};
```

필드 설명:

* `id`: 백로그 고유 ID
* `title`: 백로그 제목
* `problem`: 해결해야 할 문제 요약
* `requirement`: 요구사항
* `priority`: 우선순위
* `area`: 담당 영역
* `status`: 백로그 상태
* `relatedFeedbackIds`: 연결된 피드백 ID 목록
* `createdAt`: 생성일
* `resolvedAt`: 해결 완료일

### Backlog 상태값

```txt
todo        : 해야 할 작업
in_progress : 진행 중
done        : 해결 완료
```

## 10. RAG 설계

이 프로젝트에서 RAG는 채팅봇 형태로 노출되지 않는다.
RAG는 개발자/운영자가 백로그 자동 생성 버튼을 눌렀을 때 내부 로직으로 동작한다.

### 10.1 RAG 적용 위치

```txt
미처리 피드백
+ 기존 피드백
+ 기존 백로그
+ 제품 기능 맥락
↓
유사 사례 검색
↓
LLM 또는 생성 로직
↓
제품 백로그 생성
```

### 10.2 MVP 단계의 RAG-lite 방식

초기에는 Vector DB를 사용하지 않고, 간단한 키워드 기반 검색과 그룹핑으로 구현한다.

처리 흐름:

1. `pending` 상태의 피드백을 조회한다.
2. 피드백의 `feature`, `painPoint`, `suggestion`, `content`를 기준으로 유사도를 계산한다.
3. 유사한 피드백끼리 묶는다.
4. 기존 백로그와 유사한 내용이 있는지 확인한다.
5. 묶인 피드백 내용을 바탕으로 백로그 제목, 문제, 요구사항, 우선순위, 담당 영역을 생성한다.
6. 생성된 백로그에 관련 피드백 ID를 연결한다.
7. 관련 피드백들의 상태를 `converted`로 변경한다.

### 10.3 추후 확장 방향

추후에는 다음과 같이 확장할 수 있다.

* 피드백 내용을 Embedding으로 변환
* Vector DB에 저장
* 새 피드백과 유사한 기존 피드백 검색
* 기존 백로그와 유사한 문제 검색
* 제품 정책 문서 또는 기능 설명 문서 검색
* 검색된 문맥을 LLM 프롬프트에 포함
* LLM이 구조화된 백로그 JSON 생성

확장 예시:

```txt
현재:
키워드 기반 유사 피드백 검색

확장:
Embedding + Vector DB 기반 의미 검색
```

```txt
POST /api/feedbacks
```

```json
{
  "userId": "user-1",
  "feature": "검색",
  "painPoint": "검색 결과가 너무 많아서 원하는 상품을 찾기 어렵습니다.",
```

## 14. 면접 설명 포인트

이 프로젝트는 단순히 피드백을 저장하는 CRUD가 아니라, 고객의 불편 사항을 제품 개선 백로그로 전환하는 흐름에 초점을 둔다.

면접에서는 다음과 같이 설명할 수 있다.

```txt
Feedback2Backlog는 고객 피드백을 개발 백로그로 자동 전환하는 서비스입니다.
소비자는 피드백 작성과 처리 상태 확인만 할 수 있고,
개발자/운영자는 미처리 피드백을 하루 단위로 모아 RAG 기반으로 백로그를 생성할 수 있습니다.

단순히 피드백 하나를 요약하는 것이 아니라,
유사한 피드백을 묶어 반복적으로 등장하는 고객 문제를 찾고,
이를 제품 백로그로 전환하는 흐름을 설계했습니다.

백로그가 해결되면 연결된 피드백들의 상태가 자동으로 처리 완료로 바뀌도록 하여,
고객 피드백 수집부터 제품 개선 반영까지의 흐름을 하나의 서비스로 구현했습니다.
```
